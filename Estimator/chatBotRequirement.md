# MindfulSpend Conversational Financial Management — Claude Skill

## Skill Purpose

Implement a conversational financial management feature for the MindfulSpend application.

The feature allows users to manage financial groups, group types, actual financial transactions, budgets, and financial queries through natural-language chat and eventually voice input.

Examples:

- "Create a group called Groceries."
- "Create a group called Investments under Investment."
- "I spent ₹500 on groceries today."
- "I spent ₹2,000 on electricity yesterday."
- "Actually change that grocery expense to ₹700."
- "How much did I spend on food this month?"
- "Set my grocery budget to ₹10,000 this month."
- "Why did I spend more this month?"
- "Will I exceed my budget this month?"

The implementation MUST be divided into three independently executable project sections:

1. UI Project
2. Backend Project
3. MultiModelRouter Project

The entire feature is one product capability, but each project must be implemented independently and safely.

---

```text
  └── Tool / Function Calling Support
```

## Core architectural rule

The MultiModelRouter MUST remain application-agnostic and preferably stateless.

The Router is responsible for:

- Selecting an AI model
- Applying model fallback
- Managing model quotas
- Managing token limits
- Calling model providers
- Supporting structured tool/function calling
- Returning normalized AI responses

The Router MUST NOT be responsible for:

- User authentication
- User authorization
- JWT validation
- Conversation persistence
- Financial data persistence
- MongoDB access
- Group management
- Actual transaction management
- Business rules

Those responsibilities belong to the MindfulSpend Backend.

# 2. Security Principles

Financial data is sensitive.

## 2.1 Authentication

Every chat request MUST be authenticated using the existing MindfulSpend authentication mechanism.

The backend MUST determine the user identity from the authenticated JWT.

Never trust:

- userId supplied by the frontend
- userId generated by the LLM
- userId supplied inside a tool call
- user identity inferred from conversation text

The backend must derive:

```text
authenticatedUserId = JWT.sub
```

---

## 2.2 Conversation Ownership

Every conversation belongs to exactly one user.

The backend MUST verify:

```text
conversation.userId === authenticatedUserId
```

before loading or modifying conversation data.

A user MUST NOT be able to access another user's conversation by changing `conversationId`.

---

## 2.3 Tool Authorization

The LLM must never directly access MongoDB.

Correct flow:

```text
LLM
  │
  ▼
Structured Tool Call
  │
  ▼
Backend Tool Executor
  │
  ├── Validate arguments
  ├── Get authenticated userId
  ├── Verify ownership
  ├── Apply business rules
  └── Execute service method
```

The `userId` used by backend services MUST come from authentication context, not from the LLM.

---

## 2.4 Financial Operations

For destructive or ambiguous financial operations, confirmation should be required.

Examples:

- Delete transaction
- Bulk update
- Large financial amount
- Ambiguous category
- Ambiguous date
- Ambiguous transaction to update

Represents a user-specific financial category/group.

Examples:

```text

```

A group may belong to a Group Type depending on the existing domain model.

The implementation must reuse existing group/category structures wherever possible.

## 3.3 Actual

Represents an actual financial transaction or recorded financial value.

Examples:

```text

₹2,000 Electricity
₹85,000 Salary
₹10,000 SIP
```

The exact existing database entity must be inspected before introducing a new `Actual` entity.

If the project already uses `expense`, `transaction`, `entry`, or another equivalent concept, use the existing domain model.

Represents a persistent chat session.

Suggested structure:

```text

{

  createdAt,
  updatedAt
```

Represents an individual conversation message.

Suggested structure:

```text

{
```

```text
  toolName,
  arguments,
  result,
  status,
  createdAt
}
```

This may initially be stored only in conversation messages if the existing architecture does not require a separate collection.

# 5. Tool Design Rules

Every tool must have:

- Clear name
- Description
- Strict JSON schema
- Required parameters
- Optional parameters
- Validation rules
- Expected response format

Example:

```text
add_actual
```

Conceptually:

```json
{
  "amount": 500,
  "currency": "INR",
  "groupId": "group_123",
  "date": "2026-07-21",
  "description": "Groceries"
}
```

The actual schema MUST be adapted to the existing backend data model.

Do not invent fields that are not supported by the database.

---

# 6. Conversation Context Strategy

Do not blindly send the complete conversation history to the LLM indefinitely.

The backend should eventually support:

```text
Conversation
  │
  ├── Conversation Summary
  │
  ├── Recent Messages
  │
  └── Relevant Financial Context
```

Example:

```text
Summary:
User is managing July household expenses.

Recent Messages:
User: I spent ₹500 on groceries.
Assistant: Added ₹500 to Groceries.

Relevant Context:
Transaction ID: tx_123
Amount: ₹500
Group: Groceries
Date: 2026-07-21
```

Then:

```text
User:
Actually change that to ₹700.
```

The AI can call:

```text
update_actual
```

with:

```json
{
  "transactionId": "tx_123",
  "amount": 700
}
```

The backend remains responsible for validating transaction ownership.

## B.7 Group Resolution

If the user says:

```text
I spent ₹500 on groceries.
```

The AI may return:

```text
groupName = "Groceries"
```

The backend should resolve the group using the authenticated user's groups.

If exactly one match exists:

```text
Proceed
```

If zero matches:

```text
Ask user whether to create group
```

If multiple matches:

```text
Ask user to select group
```

Do not let the AI invent a group ID.

---

## B.8 Actual Resolution

The backend should use conversation context to identify the relevant transaction.

If exactly one transaction is clearly referenced:

```text

```

If ambiguous:

```text
Ask clarification
```

```text

₹500 at 10:00 AM
₹700 at 6:00 PM

```

## C.6 Important Execution Model

```text

  ↓

```

The router is a transport/orchestration layer.

## Definition of Done

The complete feature is considered implemented when a user can:

1. Open MindfulSpend chat.
2. Create a conversation.
3. Send natural-language messages.
4. Create and manage groups using chat.
5. Add actual financial transactions using chat.
6. Update actual transactions using conversational context.
7. Query financial data through chat.
8. Receive clarification when the request is ambiguous.
9. Confirm sensitive or destructive operations.
10. Use the same pipeline for future voice input.
11. Have all operations securely scoped to their authenticated account.
12. Use the existing MultiModelRouter without coupling financial business logic into it.

The implementation must preserve clean separation:

```text
UI
    ↓
MindfulSpend Backend
    ↓
MultiModelRouter
    ↓
LLM Provider

Financial business logic:
MindfulSpend Backend

AI routing logic:
MultiModelRouter

User interaction:
MindfulSpend UI
```

This separation is mandatory.
